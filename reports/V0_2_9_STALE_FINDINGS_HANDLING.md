# v0.2.9 Stale Findings Handling

**Date:** 2026-05-22  
**Scope:** Compliance API/UI handling for current, stale, legacy and superseded findings  
**Branch:** `feature/v0.2.9-stale-findings-handling`

## Objective

Separate current compliance results from older findings without deleting history or evidence. The UI should default to operationally relevant findings while keeping historical data accessible.

## Implemented Concepts

- `current`: finding generated by the current compliance engine/parser and belonging to the latest job for its device.
- `stale`: finding has version metadata, but the version does not match the current engine/parser.
- `legacy`: finding has no version metadata. This covers pre-v0.2.9 jobs.
- `superseded`: finding was generated by the current version, but a newer job exists for the same device.

## Versioning

New compliance findings store version metadata in `metadataJson`:

- `complianceEngineVersion`
- `parserVersion`
- `parserVersions.interface`

New discovery snapshots include:

- `parserVersion`
- `parserVersions.interface`

No database migration was required and no historical data was deleted.

## API Changes

Existing endpoints remain compatible and gained optional filters:

- `GET /api/compliance-findings`
  - `latestJobOnly=true`
  - `freshness=current|stale|legacy|superseded|all`
- `GET /api/compliance-findings-groups`
  - `latestJobOnly=true`
  - `freshness=current|stale|legacy|superseded|all`

New endpoint:

- `GET /api/compliance-findings-freshness-summary`

Summary includes:

- current findings count
- stale findings count
- legacy findings count
- superseded findings count
- latest job per device
- current engine/parser versions

## UI Changes

`/compliance` now:

- Defaults to `latestJobOnly=true`.
- Adds `Incluir histórico` toggle.
- Displays freshness counters.
- Shows alert when stale/legacy findings exist.
- Adds freshness badge per finding:
  - Atual
  - Stale
  - Legado
  - Substituído
- Keeps RBAC and read-only compliance behavior unchanged.

## Selftest Coverage

`tools/stale-findings-selftest.mjs` verifies:

- authenticated API access
- latest job map by device
- `latestJobOnly=true` returns findings only from each device's latest job
- historical findings remain accessible with `freshness=all`
- `freshness=current` returns only current findings
- `freshness=legacy` returns only legacy findings
- summary exposes freshness counts and latest jobs

## Risks

- Existing historical jobs remain visible when history is included, including pre-hotfix false positives.
- Since no migration was added, freshness is computed at read time from metadata and latest job IDs.
- Current endpoint limit behavior remains in place; very large historical sets may still require future pagination.
